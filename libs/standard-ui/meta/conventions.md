# @no-comply/standard-ui — Conventions

> This file extends [no-comply libs conventions](../../../conventions/no-comply-libs.md).

## File Organization

Every domain follows the same directory structure:

```
src/<domain>/
  index.ts                   — Barrel: re-exports components + mixins + types
  types.ts                   — (optional) domain-level types

  components/
    <Component>/
      <Component>.tsx        — SolidJS component (thin JSX wrapper)
      create<Component>.ts   — Logic factory (pure composition)
      types.ts               — Props + API types
      constants.ts           — Prop keys + registration token
      index.ts               — Re-export
      <Component>.module.scss — (optional) CSS module

  mixins/
    <Mixin>/
      create<Mixin>Mixin.ts  — Mixin factory
      types.ts               — Mixin props + API types
      constants.ts           — Prop keys
      index.ts               — Re-export
      <Mixin>Mixin.module.scss

  contexts/   (optional)
  providers/  (optional)
  controllers/(optional)

  private/
    contexts/    — (e.g., theme mode/surface contexts)
    constants/   — (e.g., theme settings)
    controllers/ — (e.g., menu internals)
```

### Barrel exports

Each domain's `index.ts` re-exports only public symbols (components, mixins, types). Private modules in `private/` are never re-exported.

The package root `src/index.ts` uses a barrel pattern generated by `@index` directives:

```ts
// @index(['./*.{ts,tsx}', './!(private|parts|functions)*/index.{ts,tsx}'], f => `export * from '${f.path.replace(/\/index$/, '')}';`)
export * from './action';
export * from './content';
// ... one line per domain
```

## Component Pattern

Every component follows this consistent structure:

```tsx
import { type ClosedTagProps, combineProps } from '@no-comply/solid-primitives';
import { type ParentComponent, splitProps } from 'solid-js';
import { Dynamic } from 'solid-js/web';

type Props = ClosedTagProps & ComponentSpecificProps;

export const ComponentName: ParentComponent<Props> = props => {
  const [locals, $others] = splitProps(props, COMPONENT_PROPS);
  const { $root } = createComponentLogic(locals);
  const $ = combineProps($others, $root);
  return <Dynamic {...$} />;
};
```

Key elements:

1. **Props intersection** — `ClosedTagProps` (intrinsic DOM props) + domain-specific props
2. **splitProps** — isolates domain props, rest pass through to DOM
3. **Factory call** — `create*()` returns a `{ $root }` API object
4. **combineProps** — merges intrinsic and computed props
5. **Dynamic** — renders the appropriate HTML element (defaults to `div` or semantic element)

## Mixin Pattern

Every mixin follows this consistent structure:

```ts
const [locals, expose, compose] = createExposable($TOKEN, props);

// Compose child mixins (optional)
const { $root: $childRoot } = compose(createChildMixin(locals));

// Compute styles
const classList = createClassList(styles, () => ['ClassName', `variant-${variant()}`]);

// Merge roots and expose
return exposeAPI(expose, '$root', {
  $root: combineProps($childRoot, $styleRoot),
  extraAccessor: () => value,
});
```

Key elements:

1. **createExposable** — creates exposable state with `compose` for child mixins
2. **createClassList** — builds conditional class string from SCSS module styles
3. **combineProps** — merges child mixin roots with computed style props
4. **exposeAPI** — exposes the mixin API through SolidJS context

## Styling Conventions

- **CSS Layers** — Declared in `layers.css`: `reset` → `composables` → `theme` → `lib`.
