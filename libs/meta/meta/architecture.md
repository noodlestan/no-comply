## Project Context

@no-comply/meta defines the entity type system and query service for the no-comply metadata extraction pipeline. It provides typed entity definitions (Component, Context, Controller, Mixin, Module, Provider, Service) and a service factory that indexes extracted metadata for fast lookup.

**Consumes:** `@purrception/primitives` (base entity types), `@purrception/lang-ts` (TypeScript AST declaration types). **Provides to:** `@no-comply/meta-extract` (entity type definitions), `@no-comply/standard-ui-demo` (meta service for querying metadata at runtime).

**API categories:** Entity type definitions (7 concrete entity shapes), service factory (`createNoComplyMetaService`), query API (`NoComplyMetaAPI` with package/module/entity lookups).

## Design Principles

### Type-Safe Entity Composition

- **Entity composition via intersection types** — `NoComplyEntityPartial & EntityDataBase` shapes built by intersecting base types from `@purrception/primitives`. Each entity adds domain-specific fields (e.g., `ComponentEntityData` adds `ComponentDeclaration` references) without class hierarchies.

- **Discriminated union on type field** — All entities carry a `type` string discriminator (`'component'`, `'context'`, etc.), enabling exhaustive matching and type-safe narrowing in consumers.

### Static Minimal Dependencies

- **Single runtime dependency** — Only `@purrception/lang-ts` at runtime. No SolidJS import in source despite the peer declaration; types reference Solid-aware metadata but the library is framework-agnostic at runtime.

- **Zero outbound from primitives** — `@purrception/primitives` has no dependencies, making circular imports structurally impossible for the base types this package extends.

### Service as Factory

- **Factory returns typed plain object** — `createNoComplyMetaService` accepts an entity array and returns `NoComplyMetaAPI`, a plain object with bound query methods. No classes, no singletons, no global state.

- **Index built once, queried many** — The 3-level index (`package > type > name`) is constructed at service creation time from the flat entity array. All subsequent queries are O(1) map lookups.

### Encapsulation

- **Private indexer** — `src/private/indexEntities` is never re-exported. Consumers depend only on the service factory and API types.

- **Barrel with codegen** — Root `index.ts` uses `// @index` codegen that skips `private/` directories.

```
└── private/

```

## Main Patterns

### Entity Types

Seven concrete entity types share a common structure: each directory contains `types.ts` (partial + data type) and `helpers/` (type guard). Entity-specific resolvers (e.g., `resolveComponentDeclaration`) live in entity `helpers/` and are consumed by `@no-comply/meta-extract`, not this package.

### Service Layer

`createNoComplyMetaService` is the sole public entry point. It accepts pre-extracted `NoComplyEntityData[]` and returns a read-only query API. The service does not fetch, parse, or transform — it indexes and retrieves.

### Shared Entity Helpers

`src/entities/helpers/` provides cross-entity utilities (`getEntityFunctions`, `getEntityTypes`) that extract declaration subsets from entity symbol arrays.

## Layers

- **Layer 0** — `@purrception/primitives`: `EntityDataBase`, `EntityDataBasePartial` (external, zero deps)
- **Layer 1** — `@purrception/lang-ts`: Declaration types, JsDoc helpers (external, only runtime dep)
- **Layer 2** — Entity definitions: `NoComplyEntityPartial`, 7 concrete entity types, type guards
- **Layer 3** — Private indexer: `indexEntities` builds 3-level Map from flat array
- **Layer 4** — Service layer: `createNoComplyMetaService` factory, `NoComplyMetaAPI` interface

Import direction is strictly upward (Layer 0 → 4). No circular dependencies.

## External Dependencies

### Runtime

- **@purrception/lang-ts** (`0.0.11`) — TypeScript AST declaration types (`ComponentDeclaration`, `FunctionDeclaration`, etc.) and JsDoc helpers. The only runtime dependency.

### Peer

- **solid-js** (`^1.9.5`) — Declared in `package.json` but not imported in source. Types reference Solid-aware metadata shapes; the library itself is framework-agnostic.

### Dev

- **vite**, **vite-plugin-solid**, **vitest** — Build tooling and test runner.

### Tradeoffs

- **Single runtime dep** — Minimal bundle, fast installs, but tightly coupled to `@purrception/lang-ts` for all declaration types.
- **Static index at creation** — Fast queries, no re-computation, but entity set is immutable after service creation.
- **No SolidJS import** — Framework-agnostic at runtime despite peer declaration; types may reference Solid shapes but source never imports `solid-js`.
