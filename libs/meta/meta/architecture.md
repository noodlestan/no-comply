## Design Principles

1. **Composition over inheritance** — Entity types are composed via intersection types (`NoComplyEntityPartial & EntityDataBase & {...}`). No class hierarchies.

2. **Separation of concerns** — Data shapes (entities) are separated from operations (helpers) which are separated from querying (service layer) which is separated from internal implementation (private module).

3. **Minimal dependencies** — Only one runtime dependency (`@purrception/lang-ts`). SolidJS is a peer dependency, not a direct import. Ships TypeScript source directly (`"main": "./src/index.ts"`).

## Layering

```
Layer 0: @purrception/primitives (external)
  EntityDataBase, EntityDataBasePartial, DocsData, LanguageDeclaredSymbol
         ↓
Layer 1: @purrception/lang-ts (external, only runtime dependency)
  Declaration types (TypeDeclaration, FunctionDeclaration, ComponentDeclaration, etc.)
  JsDoc types and helpers
         ↓
Layer 2: @no-comply/meta — Entities layer
  NoComplyEntityPartial, NoComplyEntityData
  7 concrete entity types (Component, Context, Controller, Mixin, Module, Provider, Service)
  Entity-specific type guards and resolvers
         ↓
Layer 3: @no-comply/meta — Private indexer (src/private/)
  indexEntities — transforms NoComplyEntityData[] into a 3-level Map (package > type > name)
         ↓
Layer 4: @no-comply/meta — Service layer (src/createNoComplyMetaService.ts)
  createNoComplyMetaService — factory that returns NoComplyMetaAPI
  NoComplyMetaAPI — public query interface
```

## Dependency Flow

```
@purrception/primitives  ←  @purrception/lang-ts  ←  @no-comply/meta

@no-comply/meta internal:
  src/entities/  →  src/private/indexEntities.ts  →  src/createNoComplyMetaService.ts  →  src/index.ts
```

- No circular dependencies within the package.
- No dependency on other `@no-comply` packages.
- All dependencies flow inward-to-outward (primitives → lang types → entity definitions → indexing → service).
