## Naming Conventions

| Convention           | Pattern                                           | Examples                                          |
| -------------------- | ------------------------------------------------- | ------------------------------------------------- |
| Controller factories | `createAria<PascalCaseRole>`                      | `createAriaAlert`, `createAriaPressable`          |
| Props types          | `Aria<PascalCaseRole>Props`                       | `AriaAlertProps`, `AriaPressableProps`            |
| API types            | `Aria<PascalCaseRole>API`                         | `AriaAlertAPI`, `AriaPressableAPI`                |
| Generic API types    | `AriaGeneric<PascalCase>API`                      | `AriaGenericRegionAPI`, `AriaGenericPressableAPI` |
| Props constants      | `ARIA_<SCREAMING_SNAKE>_PROPS`                    | `ARIA_ALERT_PROPS`, `ARIA_PRESSABLE_PROPS`        |
| Return keys          | `$`-prefixed (dollar sign)                        | `$root`, `$label`, `$description`                 |
| File names           | camelCase (files), PascalCase (factory functions) | `createAriaRegion.ts`                             |

## File Organization

### Per-controller 4-file structure

```
controllers/<name>/
├── index.ts         # Barrel: re-exports types, factory, constants
├── types.ts         # Props + API type definitions
├── createAria<Name>.ts  # Factory function implementation
└── constants.ts     # definePropKeys invocation (ARIA_*_PROPS)
```

### Barrel convention

- Each module directory has an `index.ts` that re-exports all sibling non-index files.
- Pattern: `export * from './types'` + `export * from './createAria<Name>'` + `export * from './constants'`.
- Top-level `src/index.ts` re-exports the four module barrels: `attributes`, `role`, `tag`, `controllers`.

### SCSS in separate tree

- Mixins live in `scss/mixins/`, not co-located with controllers.

## API Design Patterns

1. **Factory pattern over classes** — All controllers are functions, not classes. They return reactive object literals, not instances.

2. **Overloaded signatures for static vs dynamic roles** — Factories have two overloads: one with `staticRole` (fixed role → narrowed return type), one without (dynamic role → generic return type).

3. **Props-in, API-out** — Input is a plain `Props` object; output is a reactive `API` object whose properties are SolidJS accessors generated by `computedProps`.

4. **Composition via delegation** — Higher-level controllers compose lower-level ones internally. Output merging via `combineProps`.

5. **Return shape convention** — Every controller returns at minimum `{ $root }`. Controllers with labelling also return `$label`, `$description`, `hasLabel`.

6. **Prop key constants** — Each controller exports an `ARIA_*_PROPS` array via `definePropKeys<T>()([...])` for runtime prop filtering.

7. **All named exports** — No default exports anywhere.

8. **No JSX output** — Pure reactive prop factories, not SolidJS components.

## Typing Conventions

1. **Intersection types for API composition** — e.g., `AriaAlertAPI['$root']` extends `AriaRegionAPI<'alert'>['$root']` via intersection.

2. **Omit + intersection to override base fields** — e.g., `AriaMenuAPI = Omit<AriaLabelledAPI, '$root'> & { $root: { ... } }` to narrow `$root`.

3. **No inheritance** — Composition via structural typing (intersections), never class extends.

4. **Generic type parameters for role narrowing** — e.g., `AriaRegionAPI<T extends AriaRoleName>`, `AriaPressableAPI<T extends PressableRoleName>`.

5. **Literal types for static values** — `'aria-live': 'assertive'` (literal, never `string`).

6. **`as const` assertions** — Applied to emitted static attribute objects.

7. **Union types over enums** — Role names, tag names, and attribute values are union types, not TypeScript enums.
