## Project Context

@purrception/lang-ts provides lightweight TypeScript AST type definitions, node helpers, and a resolution engine for the Purrception ecosystem. It defines the structural vocabulary that downstream extractors (`lang-ts-extract`) use to represent and resolve TS types, declarations, and JSDoc data.

**Consumes:** `@purrception/primitives` — base entity types (`EntityDataBase`, `LanguageDeclaredSymbol`, `DeclaredSymbol`).

**Provides to:** `@purrception/lang-ts-extract` — domain types (`TypeExpressionNode`, `FunctionDeclaration`, `TypeDeclaration`, `JsDocData`), helpers (`createNode`, `isNode`), and the resolution engine (`resolveExpression`, `resolveTypeDeclaration`).

**API categories:** Type definitions (AST nodes, declarations, JSDoc), node helpers (create/check), resolution engine (expression + declaration resolvers), and language constants.

## Design Principles

### Type System

- **Discriminated unions with `kind`** — Every type expression and declaration uses a `kind` string discriminator enabling switch-based dispatch and exhaustiveness checking.

- **Separation of data from behavior** — Types live in `types.ts` files. Operations live in separate function modules. No mixing.

- **Immutable transformations** — Resolvers produce new objects via spread; no in-place mutation. Predictable and safe for recursive resolution.

### Resolution

- **Switch-based dispatcher** — `resolveExpression` dispatches on `kind` to per-kind resolvers. Simple, explicit, and type-safe.

- **Thin resolution functions** — Each resolver performs one AST-to-model conversion with no side effects. Resolvers are pure compositions of lower-level helpers.

- **Recursive with immutability guard** — Deeply nested types are resolved via immutable spreads and recursion. Performance cost is accepted for correctness.

### Extensibility

- **Closed for external extension** — No plugin system. New type expression kinds require adding to the `kind` union, defining the type, adding a case in `resolveExpression`, and writing a resolver.

- **Open within the module** — New resolvers, normalizers, and operators can be added within `resolve/` without modifying other layers.

## Structure

```
src/
├── constants.ts                       — Language identifiers (PurrceptionLanguageId, LanguageName)
├── index.ts                           — Public barrel re-export
├── declaration/
│   ├── types.ts                       — Declaration discriminated union types
│   └── helpers/ // Component, Function, Interface, Type + 3 more guards
├── jsdoc/
│   ├── types.ts                       — JsDocData, FunctionJsDocData, JsDocTags
│   └── helpers/ // getFirstTagValue
├── node/
│   ├── types.ts                       — TypeExpressionNode discriminated union (18 kinds)
│   ├── constants.ts                   — BUILTIN_TYPES and BUILTIN_GLOBALS
│   └── helpers/ // create*Node, is*TypeNode, isBuiltInToken
└── resolve/
    ├── types.ts                       — ResolveTypeContext
    ├── createResolveTypeContext.ts    — Factory for resolution context
    ├── resolveTypeDeclaration.ts      — Top-level entry point
    ├── declarations/ // resolveInterface, resolveTypeExpression
    └── expressions/
        ├── resolveExpression.ts       — Dispatcher (switch on kind)
        ├── kinds/ // per-kind resolvers
        ├── normalize/ // normalizeUnion, normalizeObjectLiteral
        └── operators/ // resolveOperatorKeyOf
```

The five modules form a strict layering: `constants` → `jsdoc` → `node` → `declaration` → `resolve`. Each depends only on layers below it.

## Main Patterns

- **Type Expression Nodes** — 18 discriminated kinds (`TypeRefNode`, `IntersectionNode`, `UnionNode`, `ObjectNode`, etc.) forming the core AST vocabulary. Created via `create*Node` helpers, checked via `is*TypeNode` guards.

- **Declaration Wrappers** — `FunctionDeclaration`, `ComponentDeclaration`, `InterfaceDeclaration`, `TypeDeclaration` wrap type expression nodes with additional metadata (JSDoc, heritage, members).

- **Resolution Engine** — `resolveExpression` dispatches to per-kind resolvers. `resolveTypeDeclaration` wraps declaration-level resolution. Heritage chains are flattened; object members are merged.

- **JSDoc Data Model** — `JsDocData` and `FunctionJsDocData` capture parsed JSDoc tags. `getFirstTagValue` extracts values from multi-valued tags.

## Layers

Import direction is strictly downward: `constants` → `jsdoc` → `node` → `declaration` → `resolve`. No module depends on a module at the same or higher layer. The `resolve` layer is the only consumer of all other layers. No circular dependencies exist.

## External Dependencies

- **@purrception/primitives** (`0.0.11`) — Base entity types consumed by `declaration` and `resolve` layers.
- **typescript** (`^5.9.3`) — Compiler API used downstream by `lang-ts-extract`, not directly by this package at runtime.

### Tradeoffs

- **18 type expression kinds** — Covers the vast majority of TS types without modeling every possible AST node.
- **Switch-based dispatcher** — Simple and explicit, but harder to extend externally.
- **Immutable spreads + recursion** — Safer and predictable; potential performance cost for deeply nested types.
- **No internal plugin system** — Stays focused and simple; new kinds require upstream changes.
