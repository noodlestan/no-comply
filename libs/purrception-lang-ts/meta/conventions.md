## File Organization Patterns

1. **Module-per-concept**: Each conceptual domain has a top-level directory under `src/` (`declaration/`, `jsdoc/`, `node/`, `resolve/`).
2. **Barrel exports via `@index`**: Each module has an `index.ts` using generated barrel exports (`// @index(['./*.{ts,tsx}', ...])`).
3. **Types/logic separation**: Types defined in `types.ts`; functions in individual files or `helpers/` subdirectories.
4. **Recursive `index.ts` pattern**: Parent modules re-export from children (e.g., `src/index.ts` → `src/declaration/index.ts` → `src/declaration/helpers/index.ts`).
5. **Sub-resolver isolation**: Complex logic (resolve expressions) further decomposed into `kinds/`, `normalize/`, `operators/` subdirectories.

## API Design Patterns

1. **Discriminated unions as primary abstraction** — `TypeExpressionNode`, `Declaration`, `DeclarationKind`, `TypeExpressionKind` use `kind: string` discriminator.
2. **Type guards for narrowing** — Every discriminated union member has a corresponding `is*` type guard function.
3. **Factory functions for constructors** — `createPrimitiveNode`, `createTypeRefNode` for simple node creation.
4. **Context object pattern** — `ResolveTypeContext` bundles entity metadata, import resolution, parameter substitutions, and circular-reference tracking.
5. **Immutable transformations** — All resolvers return new objects via spread (`{...node, ...}`); no mutation.
6. **Switch dispatch** — `resolveExpression` uses a switch on `kind` to delegate to specialized resolvers.
7. **Error handling** — Unknown references and invalid expressions produce `console.warn` and return original node. Circular references throw `Error`.

## Typing Conventions

- **Preference for `type` over `interface`** — All type definitions use `type` aliases.
- **Constrained generics** — Generics constrained with `extends` for discriminated union tagging (`TypeExpressionBase<K extends TypeExpressionKind>`).
- **`Record<string, T>` for dictionaries** — Object members use `Record<string, ObjectLiteralTypeMember>`.
- **Intersection for mixin** — `DeclarationBase<K>` = `LanguageDeclaredSymbol<Id> & JsDocData & { ... }`.
- **Optional fields** — Marked with `?` liberally (e.g., `description?`, `generic?`, `_source?`).
- **Readonly modifier** — Used sparingly on object member signatures.
