## Project Context

The demo application for the `@no-comply/standard-ui` component library. It provides interactive documentation, live component playgrounds, API reference pages, and design token browsing — all rendered client-side with no runtime data fetching.

**Consumes:** All `@no-comply/*` libraries (standard-ui, solid-contexts, solid-composables, solid-accessibility, solid-primitives, solid-dev-tools, meta), `@purrtrait/*` code rendering packages, `@purrpose/*` in-browser compilation, and `@purrception/lang-ts` for metadata types.

**Provides:** A single-page application that showcases every `@no-comply/standard-ui` component with editable examples, props documentation, and accessibility annotations.

**API categories:** Screens (route-level pages), Providers (SolidJS context wrappers), Services (app-level factories), Components (reusable UI organized by domain), Content (doc pages, examples, and factories per component).

## Design principles

### Context-based dependency injection

All services are injected through SolidJS `Context` — no global state, no singletons. Guarded hooks (`useMeta()`, `useAppServices()`) enforce provider wrapping and keep access explicit.

### Static build-time composition

Meta JSON, theme data, and component examples are bundled at build time via Vite. No runtime data fetching, no server dependency — the app is fully offline-capable.

### Convention-driven structure

Screens, templates, and content pages follow consistent directory conventions rather than config files. Route definitions are centralized in `src/app/navigation/Routes.tsx`; new screens slot in by adding a route entry.

### Private context pattern

Each provider follows the same boilerplate shape: context definition, provider component, guarded hook, and optional factory. This creates clear API boundaries at the cost of some repetition.

### In-browser compilation

Live TSX playgrounds compile user-editable code in the browser using `@purrpose/client-babel` with the SolidJS preset. This trades bundle size for interactivity.

## Main patterns

### Screens

Route-level pages (`ApiScreen`, `ComponentsScreen`, `ResourcesScreen`, `AppHomeScreen`, and 3 more) that consume providers via hooks and render templates. Each screen owns its own sub-components and pages.

### Providers

SolidJS context wrappers — `AppServices`, `CodeBlock`, `Meta`, `UIRoot`. `AppServices` creates and injects service factories; `Meta` loads and indexes extracted component metadata from `@no-comply/meta`.

### Services

Factory functions (`createAppNavigation`, `createAppStatus`) returning typed plain objects. Services are created once in `AppServicesProvider` and accessed exclusively through context.

### Components

Reusable UI organized by domain: `atoms/` (Spinner), `cards/` (Card), `code/` (CodeBlock, CodeInline, JSXRenderer), `splash/` (AppSplash, SplashBox), `theme/` (ThemeSelect).

### Content

Doc pages, examples, and factories per component. `content/types/` defines `DocsPageData`, `DocsSectionData`, `DocsItemData`. `content/factories/` builds structured doc data from static JSON imports.

### Templates

Reusable screen and page layouts: `BasePage`, `EmptyPage`, `PageContentsLayout`, `WithSidebarLayout`. Screens compose templates rather than defining their own layout.

## Layers

Import direction flows downward: Entry → App (Router + Providers) → Screens → Templates → Components. Infrastructure layer (Providers ↔ Services) sits between App and Foundation. Foundation holds `data/`, `env/`, `styles/`, and `utils/`.

Screens never import services directly — all access is through context hooks. Providers call service factories; services may import `navigation/routeFor` but nothing imports upward from services.

## External dependencies

- **SolidJS ecosystem** — `solid-js`, `@solidjs/router`, `solid-services`, `solidjs-use`, `lucide-solid` for reactive UI, routing, service utilities, composition helpers, and icons
- **@no-comply/\*** — standard-ui (component library), solid-contexts (theme/mode/surface providers), solid-composables, solid-accessibility, solid-primitives, solid-dev-tools, meta (metadata service)
- **@purrtrait/\*** — code-layout, lang-ts, client-tsx, solid-code, view-tsx for code rendering and layout
- **@purrpose/\*** — client-babel + preset-solidjs for in-browser TSX compilation
- **Vite** — Build tool and dev server with SolidJS, SVG, and top-level-await plugins
- **@solid-primitives/\*** — intersection-observer, resize-observer, storage for DOM utilities

### Tradeoffs

- **No data fetching** — Simple architecture, offline-capable, but not dynamic
- **Static meta JSON at build time** — Fast runtime, no server needed, but requires rebuild for content changes
- **In-browser TSX compiler** — Enables live playground but bundles large TypeScript/Babel dependencies
- **Heavy monorepo dependency** — Tight coupling to `@no-comply/*` ecosystem; hard to reuse outside
- **Private context pattern** — Clear API boundaries, but boilerplate-heavy (context + provider + hook + factory)
