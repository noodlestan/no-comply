## Design Principles

| Principle                              | Evidence                                                                                                                                                    |
| -------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Composition over inheritance**       | Provider nesting, template decoration (`ScreenTemplateBase` → `WithSidebar`), recursive content section nesting                                             |
| **Factory pattern for services**       | All services are factory functions (`createAppNavigation`, `createAppStatus`, `createTSXCompilerModule`, `createAppServices`) returning typed plain objects |
| **Context-based dependency injection** | SolidJS `Context` for all service injection; guarded hooks (`useXxx`) enforce provider wrapping                                                             |
| **Encapsulation via `private/`**       | Internal implementation details hidden behind barrel exports; `private/` directories never re-exported                                                      |
| **Static build-time composition**      | Meta JSON, theme data, examples bundled at build time; no runtime data fetching                                                                             |
| **Reactive primitives**                | SolidJS `Accessor<T>` pattern for all reactive values; no mutable state                                                                                     |

## Layering

```
External Libraries (@no-comply/*, @purrception/*, @purrtrait/*, @purrpose/*)
       ↓
   src/providers/  ────────→  src/services/  ──────→  src/app/navigation/
       ↓                              ↓
   src/app/ (providers wrap screens, screens consume providers via hooks)
       ↓
   src/app/screens/  →  src/app/templates/  →  src/app/components/
       ↓
   src/app/content/  (types, factories, examples used by screens)
```

## Dependency Flow

- **Providers depend on services** — `AppServicesProvider` calls `createAppNavigation`, `createAppStatus`
- **Screens depend on providers** — via `useMeta()`, `useAppServices()` hooks
- **Services depend on app navigation** — `createAppNavigation` imports `routeFor`
- **No circular dependencies** — clear downward flow through layers
- **No cross-layer skip** — screens do not directly import services; all access is through context

## Trade-offs

| Decision                       | Trade-off                                                                         |
| ------------------------------ | --------------------------------------------------------------------------------- |
| Static meta JSON at build time | Fast runtime, no server needed, but requires rebuild for content changes          |
| No data fetching               | Simple architecture, offline-capable, but not dynamic                             |
| Heavy monorepo dependency      | Tight coupling to `@no-comply/*` ecosystem; hard to reuse outside                 |
| In-browser TSX compiler        | Enables live playground but bundles large TypeScript/Babel dependencies           |
| CSS Modules + SCSS globals     | Scoped styles with shared design tokens, but adds build complexity                |
| Private context pattern        | Clear API boundaries, but boilerplate-heavy (context + provider + hook + factory) |

## Extensibility Model

1. **Plugin via Provider** — Wrap a new provider around existing ones to add capabilities
2. **Content via Factories** — New doc pages via `createDocsPageData` / `createComponentDocsPage`
3. **Theme via Data** — Add entries to `THEMES` static array
4. **Screens via Routes** — New routes added in `src/app/navigation/Routes.tsx`
5. **Meta via JSON** — Add more `@no-comply/*` meta JSON imports in `MetaProvider`
